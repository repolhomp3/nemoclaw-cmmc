## 1. Purpose

This standard defines how the organization identifies, classifies, stores, transmits, reviews, sanitizes, exports, and disposes of **derived Controlled Unclassified Information (CUI)** generated by NemoClaw on the DGX Spark platform.

This standard exists because NemoClaw processes CUI software source code and creates additional artifacts that may themselves contain, reveal, summarize, reproduce, or enable reconstruction of CUI.

---

## 2. Scope

This standard applies to all NemoClaw users, administrators, auditors, service accounts, and supporting systems that handle outputs or metadata generated from CUI processing on the DGX Spark system.

It applies to the following data classes when created from or materially informed by CUI:
- user prompts
- model responses
- retrieved code snippets
- module summaries
- vector embeddings
- conversation history
- exported reports
- cached context windows
- audit records containing code excerpts, prompts, responses, or sensitive file/object references

---

## 3. Governing Principle

**Unless explicitly reviewed and approved as sanitized, outputs and artifacts derived from CUI shall be treated as CUI.**

This includes artifacts that:
- directly contain CUI
- summarize CUI in a way that preserves sensitive meaning
- permit reconstruction of CUI content or architecture
- expose sensitive file structure, function names, code logic, vulnerabilities, or technical context

---

## 4. Classification Rules

## 4.1 Automatically treated as CUI-derived
The following shall be treated as CUI-derived by default when produced from in-scope repositories or workflows:
- responses containing verbatim code, partial code, pseudocode based on CUI code, or technical analysis of CUI code
- summaries of source files, modules, interfaces, or system behavior
- vector embeddings generated from CUI source code or CUI-derived summaries
- prompt and response logs tied to CUI analysis sessions
- session transcripts and feedback logs that reference CUI content
- audit events that include code excerpts, internal file paths, object names, or output excerpts
- downloaded or exported reports containing code-derived findings

## 4.2 Potentially non-CUI only after review
The following may be released from CUI handling requirements only after review by an authorized reviewer under approved sanitization procedures:
- high-level statistical summaries with no technical detail
- generic usage metrics with no user-sensitive or content-sensitive context
- public-facing descriptions that contain no CUI, no derived technical detail, and no sensitive architecture disclosure

## 4.3 Not automatically CUI
The following are not CUI solely by presence on the system, unless modified or combined with CUI-derived content:
- public base model weights
- public open-source test datasets used outside the boundary
- generic software package metadata that does not reveal protected content

---

## 5. Authorized Handling and Access

## 5.1 Access restrictions
Derived CUI shall only be accessible to:
- authorized users with a valid business need
- administrators performing approved security or maintenance duties
- auditors with approved read-only review scope
- service accounts required to process, store, or protect the data

## 5.2 Least privilege
Access to derived CUI must follow least privilege and role separation requirements. Developers shall not receive administrative access to logs or stores solely because they can use the application.

## 5.3 Same-level protection requirement
Derived CUI shall receive protections equivalent to the underlying CUI source unless a formal determination authorizes lesser handling.

---

## 6. Storage and Retention

## 6.1 Approved storage locations
Derived CUI may be stored only in approved locations, including:
- protected application data stores
- protected conversation history stores
- protected vector databases
- protected audit/log repositories
- approved exported evidence or reporting repositories within the CMMC boundary

## 6.2 Prohibited storage locations
Derived CUI shall not be stored in:
- personal user directories unless explicitly approved and protected
- unmanaged workstations
- public collaboration platforms
- internet-connected services outside the boundary
- removable media without approved transport and protection procedures

## 6.3 Retention
Retention periods for derived CUI artifacts shall be defined in the system retention schedule. At minimum, the retention schedule shall specify periods for:
- conversation history
- prompts and responses
- vector embeddings
- summaries and generated reports
- audit logs
- temporary files and caches

Where retention periods are not yet finalized, derived CUI shall be retained only as long as operationally required and shall remain subject to full CUI handling requirements.

---

## 7. Transmission and Export

## 7.1 Internal transmission
Derived CUI may only be transmitted over approved internal paths protected by approved cryptographic and enclave boundary controls.

## 7.2 Export restrictions
Derived CUI shall not be exported from NemoClaw except through approved workflows. Any export capability, including download, print, copy, clipboard transfer, report generation, or API transfer, shall be:
- documented
- technically controlled where feasible
- attributable to an authenticated user
- logged with timestamp and object details

## 7.3 Prohibited destinations
Derived CUI shall not be transmitted to:
- public websites
- unmanaged endpoints
- external AI services
- personal email or messaging tools
- third-party cloud storage outside the approved boundary

---

## 8. Sanitization and Release

## 8.1 Sanitization requirement
Before a derived artifact is treated as non-CUI or released outside controlled handling, an authorized reviewer shall determine that the artifact no longer contains or meaningfully reveals CUI.

## 8.2 Sanitization criteria
A derived artifact shall not be considered sanitized if it contains or reveals:
- verbatim or near-verbatim code
- architecture details that remain sensitive
- internal interfaces, paths, or identifiers
- vulnerability details tied to protected systems
- logic summaries that enable re-creation of protected implementation details

## 8.3 Review record
Sanitization decisions shall be documented with:
- reviewer identity
- date and time
- object reviewed
- disposition decision
- basis for sanitization or continued control

---

## 9. Logging Requirements

The system shall log handling events for derived CUI where technically feasible, including:
- generation of outputs
- downloads/exports
- administrative access to logs or stores
- changes to retention or storage policies
- sanitization/release decisions

Logging records shall themselves be treated as CUI if they contain sensitive details.

---

## 10. Disposal and Destruction

Derived CUI shall be sanitized or destroyed in accordance with approved media and data destruction procedures when no longer needed. This includes:
- deletion of temporary application artifacts
- sanitization of removable media
- secure handling of backups
- decommissioning procedures for local storage devices

For encrypted local storage, encryption alone is not a substitute for approved sanitization/disposal requirements where media is reused, transferred, or retired.

---

## 11. Roles and Responsibilities

### System Owner
- approves this standard
- ensures supporting procedures exist
- resolves classification disputes

### Security/Compliance Team
- defines review and retention expectations
- validates implementation against CMMC requirements
- reviews exceptions

### Administrators
- implement technical restrictions and storage controls
- protect logs and system stores
- enforce export and retention settings

### Users
- handle outputs as CUI unless explicitly told otherwise
- do not copy or share outputs outside approved channels
- report suspected disclosure or misclassification

### Auditors
- verify that derived artifacts are handled at the required protection level

---

## 12. Exceptions

Any exception to this standard requires documented approval by authorized management and security personnel. Exceptions shall:
- identify the specific data class
- define the duration and scope
- document compensating controls
- be logged and reviewable

## 13. Mapping to NIST SP 800-171 Rev. 2 / CMMC L2

This standard primarily supports:
- **AC.L2-3.1.1** authorized access
- **AC.L2-3.1.2** authorized functions
- **AC.L2-3.1.3** control CUI flow
- **AU.L2-3.3.1** retain audit records
- **MP.L2-3.8.1** protect media containing CUI
- **MP.L2-3.8.2** limit access to media containing CUI
- **MP.L2-3.8.3** sanitize media before disposal or reuse
- **SC.L2-3.13.16** protect confidentiality of CUI at rest

---

## 14. Implementation Notes for NemoClaw

For NemoClaw on DGX Spark, the following stores shall be treated as containing derived CUI whenever the system analyzes in-scope code:
- PostgreSQL/pgvector database
- SQLite conversation stores
- application export directories
- audit/log directories containing prompts, outputs, excerpts, or sensitive references
- generated summaries and supporting evidence bundles

If future fine-tuning on CUI is introduced, this standard must be updated to treat resulting model artifacts appropriately.
